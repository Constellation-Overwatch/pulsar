# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Pulsar is the edge sync agent for Constellation Overwatch. It runs as a single Go binary on ground control stations to auto-register fleet entities (drones, sensors, vehicles), relay MAVLink telemetry over NATS JetStream, bridge RTSP video streams, and run optional on-device YOLO detection.

## Build & Run Commands

Uses [Task](https://taskfile.dev/) runner. All commands defined in `Taskfile.yml`.

```bash
task dev                # Build and run without detection (go run ./cmd/microlith)
task build              # Build binary to bin/pulsar (no detection)
task run                # Build + run binary

task build:detection    # Build binary (detection enabled via ONNX_LIB_PATH at runtime)
task dev:detection      # Dev mode with detection (needs ONNX Runtime + ffmpeg)
task run:detection      # Build + run with detection

task setup:deps         # Install ONNX Runtime (macOS/Homebrew)
task setup:model        # Download YOLO26 ONNX from HuggingFace
task setup:all          # Full detection setup (deps + model)

task docker-build       # Build Docker image
task docker-run         # docker-compose up -d

task test-relay         # Synthetic MAVLink relay integration test (scripts/mavlink-relay.sh)
task clean              # Remove bin/ and snapshot.jpg
```

### Running Tests

```bash
go test ./...                           # All tests
go test ./pkg/services/registry/...     # Registry tests (main test suite)
go test -run TestRegister_FirstBoot ./pkg/services/registry/...  # Single test
```

## Architecture

**Single entry point:** `cmd/microlith/main.go` — bootstrap, guided first-time setup wizard, and sync loop.

**Reconciliation pattern:** Pulsar reconciles desired state (`config/fleet.yaml`, user-authored) against actual state (`config/c4.json`, machine-generated). On each boot/sync cycle it compares entity names to track UUIDs, detects drift (name, priority, status changes), creates/updates/deletes entities on the Overwatch API, and writes resolved state back to `c4.json`.

**Service modules** in `pkg/services/` start/stop independently per sync cycle:
- **registry/** — Overwatch API registration and reconciliation logic. Uses `pkg/shared/client.go` (HTTP client with 3-attempt retry). This is the core orchestration service.
- **relay/** — Per-entity MAVLink UDP listeners. Parses frames and dispatches to publisher.
- **publisher/** — NATS JetStream publishing on `constellation.telemetry.{entity_id}.{msg_type}` + KV state aggregation in `CONSTELLATION_GLOBAL_STATE` bucket.
- **video/** — RTSP server (auto-detects MediaMTX or falls back to embedded gortsplib), ffmpeg-based source bridging (publishes sources to MediaMTX), ffmpeg-based overlay rendering (annotated frames → RTSP).
- **detector/** — YOLO26 ONNX inference with object tracking via onnxruntime-purego (no CGO). Runtime-optional: disabled when ONNX_LIB_PATH is not set.

**Shared types** in `pkg/shared/`:
- `config.go` — Fleet/C4 types, YAML unmarshaling (supports `mavlink: true` and `mavlink: {port: N}`), entity type mapping (e.g., `uav` → `aircraft_multirotor`), MAVLink port resolution.
- `client.go` — Overwatch REST API wrapper (CRUD for orgs and entities).
- `network.go` — NIC priority ranking and `ADVERTISE_HOST` auto-discovery.

## Key Patterns

- **CGO-free build:** All code compiles with `CGO_ENABLED=0`. Detection uses onnxruntime-purego (dlopen at runtime), video uses ffmpeg subprocesses. No build tags needed.
- **Dual video hosts:** `RTSP_HOST` (local, for bridge/detector) vs `ADVERTISE_HOST` (external, registered with Overwatch). Registry generates advertised endpoints — never sends `fleet.yaml` video_config to Overwatch directly.
- **MAVLink port assignment:** Explicit ports reserved first, then auto-assigned ports fill sequentially from `MAVLINK_BASE_PORT` (default 14550), skipping conflicts.
- **Config hashing:** The sync loop hashes `fleet.yaml` to detect changes, re-registering and restarting services only when config actually changes.
- **Consistent log prefixes:** `[pulsar]`, `[relay]`, `[video]`, `[detector]`, `[bridge]`.
- **Context-based lifecycle:** Services accept `context.Context` for cancellation and graceful shutdown.

## Configuration

- **`.env`** — Required: `C4_API_KEY`, `C4_BASE_URL`, `C4_NATS_KEY`. Optional: `MAVLINK_BASE_PORT`, `RTSP_HOST`, `ADVERTISE_HOST`, `MODEL_PATH`, `ONNX_LIB_PATH`.
- **`config/fleet.yaml`** — Declarative fleet definition (user-authored, committed). Generated by guided setup on first run if missing.
- **`config/c4.json`** — Resolved state with UUIDs, ports, RTSP URLs (machine-generated, gitignored).

## Testing

Tests use `httptest.NewServer` with a `mockOverwatch` that simulates the full Overwatch API. Test cases cover: first boot creation, idempotent re-registration, drift detection/update, stale entity removal, re-provisioning after external deletion, adding new entities, and config hashing.

## Dependencies

Go 1.25. Key libraries: `gomavlib/v3` (MAVLink), `gortsplib/v5` (embedded RTSP), `nats.go` + `jetstream` (NATS), `yaml.v3`, `onnxruntime-purego` (ONNX inference, no CGO), `disintegration/imaging` (image processing). Runtime: `ffmpeg` (video source bridging, H264 encoding, overlay publishing), `libonnxruntime` dylib/so (detection inference).

## Go Best Practices

Follow the [Zen of Go](https://bitfieldconsulting.com/posts/go-right-way) when writing and reviewing code in this project:

### 1. Write Packages, Not Programs
Keep `main.go` minimal — delegate business logic to packages in `pkg/services/` and `pkg/shared/`. Packages should return data, not print. Avoid `panic` in library code. Maintain flat module structure to prevent import cycles.

### 2. Test Everything
Tests are a design tool. Write small, focused tests examining single behaviors through public APIs. Use `httptest.NewServer` for HTTP mocks (see `registry_test.go`). Cover: happy path, idempotent re-runs, drift detection, cleanup of stale state.

### 3. Write Code for Reading
Prioritize clarity over cleverness. Use well-chosen names that convey purpose. Comments explain *why*, not *what*. Refactor until logic is obvious. Consistent naming across the codebase (e.g., `entity` not sometimes `device`/`drone`/`asset`).

### 4. Be Safe by Default
Design types so invalid states are impossible. Leverage useful zero values. Use named constants, not magic numbers. Employ `WithX` option patterns for configuration. Keep credential handling in `.env` — never hardcode secrets.

### 5. Wrap Errors, Don't Flatten
Use `fmt.Errorf("context: %w", err)` to wrap errors with context. Define sentinel errors for callers to match with `errors.Is`. Never inspect error strings directly.

### 6. Avoid Mutable Global State
No package-level mutable variables. Pass dependencies explicitly through function parameters or struct fields. Create new instances rather than relying on global defaults (e.g., `http.NewServeMux()` not `http.DefaultServeMux`).

### 7. Use Structured Concurrency Sparingly
Only introduce goroutines when necessary. Confine goroutines to their creation scope. Ensure all goroutines terminate before the parent function exits using `sync.WaitGroup` or `context.Context`. Accept unidirectional channels.

### 8. Decouple Code from Environment
Read environment variables only in `main.go`. Pass config values as function parameters or struct fields to services. Use `go:embed` for bundled assets. Avoid hardcoded paths — use config or env vars.

### 9. Design for Errors
Always check and handle errors. Implement retry logic for transient failures (see `client.go` 3-attempt retry). Reserve `panic` exclusively for unrecoverable internal bugs. Provide helpful error messages.

### 10. Log Only Actionable Information
Use our `logger` package (zap-based), not `log.Printf`. Log errors requiring user action. Use structured fields (`logger.Infof`). Never log secrets or credentials. Maintain consistent log prefixes: `[pulsar]`, `[relay]`, `[video]`, `[detector]`, `[bridge]`.
